보안 강점

한 줄 요약 — 라우팅과 RLS 소유권 모델은 견고하고 포트폴리오·상점 결제는 서버에서 원자적으로 처리돼 안전합니다. 다만 채점이 브라우저(Pyodide)에서 이뤄지고 그 점수를 클라이언트가 그대로 DB에 기록하기 때문에 포인트 경제가 위조 가능합니다 — 이것이 최우선 수정 대상입니다.

01 검증

권한 기반 라우팅 검증

모든 보호 라우트는 RequireRole / RequireAuth 래퍼를 거칩니다. 미로그인 상태의 리다이렉트는 Playwright로 실제 실행해 확인했고( 표시), 역할 불일치 리다이렉트는 세션 계정이 없어 코드 레벨로만 검증했습니다(◐ 표시).

라우트가드대상미로그인 결과
/Home공개Landing 렌더
/dashboardRequireRole teacher교사/ 로 리다이렉트
/classes · /students · /problems · /shopRequireRole teacher교사/ (코드 검증)
/classes/:id/liveRequireRole teacher교사/ (코드 검증)
/studentRequireRole student학생/ 로 리다이렉트
/myclass · /practice/* · /student/shopRequireRole student학생/ (코드 검증)
/practice/typingRequireAuth공통/ (코드 검증)
/solve/:problemIdRequireAuth공통/ 로 리다이렉트
*—공개NotFound 렌더
역할 분기 로직은 견고합니다. RequireRole은 역할 불일치 시 각자 홈(teacher→/dashboard, student→/student)으로 보내고, 프로필 행이 없는 세션(role === null)은 무한 리다이렉트 대신 OrphanedSession이 강제 로그아웃 처리합니다. 주의: 라우트 가드는 화면 전환만 막습니다 — 실제 데이터 보호는 전적으로 RLS에 의존하므로, 아래 RLS 취약점이 그대로 실질 위험이 됩니다.

02 흐름

사용자 여정 · 마찰 지점

Teacher

교사 흐름

  1. 가입 시 선생님 코드 입력 → claim_teacher로 승격
  2. 문제 작성(순서도 DSL · 스타터/테스트 코드 · 배점)
  3. 반 생성 → 학생 등록 → 문제 배정
  4. 대시보드에서 제출·라이브 코드·타자 기록 확인
  5. 상점 상품 등록, 구매 요청 승인/거절
  6. 포트폴리오 제출본에 범위 지정 피드백

이메일 확인이 켜진 프로젝트라면 가입 직후 세션이 없어 교사 승격이 로그인 이후로 미뤄짐 — 첫 로그인에서 학생 화면이 잠깐 노출될 수 있음.

Student

학생 흐름

  1. 아이디로 가입(합성 이메일 id@flowpy.local)
  2. 내 반의 배정 문제 열람
  3. 순서도를 보며 코드 작성 · Pyodide 실행
  4. 제출 → 브라우저에서 채점 → 만점 시 포인트
  5. 타자 연습 · AI 연구소로 추가 포인트
  6. 상점에서 포인트로 상품 구매 요청

채점이 클라이언트에서 끝나므로 테스트 기대값이 학생에게 그대로 노출됨 — 숨김 테스트 케이스 구성이 불가능(정답 열람 가능).

03 보안

RLS · 무결성 취약점 (심각도순)

SEC-1

클라이언트 채점 점수로 포인트 위조

Critical
근거
채점은 브라우저 Pyodide에서 수행되고 submitSolution()이 클라이언트가 계산한 score/max_score를 그대로 submissions에 삽입. RLS insert 체크는 user_id = auth.uid()와 문제 is_published만 확인.
영향
학생이 개발자도구에서 문제를 풀지 않고 score = max_score 행을 삽입하면 트리거 award_points_on_submission가 자동으로 포인트 지급 → 상점에서 실물(간식 쿠폰·숙제 면제권)로 교환. 포인트 경제 전체가 위조 가능.
수정
서버 채점으로 이전: 엣지 함수/RPC가 테스트를 실행하거나 기대 출력을 서버에만 보관해 검증. 최소한 클라이언트 자기보고 점수로는 포인트를 지급하지 않도록.
SEC-2

타자 포인트 파밍 (작업 증명 없음)

High
근거
complete_typing_practice는 보상 액수는 서버가 고정하지만 실제 타자가 일어났다는 증거를 요구하지 않음. race_live의 p_won은 클라이언트 값이며 무검증(+5). 유일한 제약은 모드별 10초 쿨다운.
영향
스크립트로 RPC를 반복 호출해 분당 약 60~90P 채굴 가능. (ai_competition은 매치 참가를 검증해 안전하나 race_live/ghost/practice는 미검증.)
수정
보상을 서버가 만든 결과 행에 바인딩(ai_competition 패턴 재사용) 또는 일일 상한 설정.
SEC-3

교사 간 학생 PII 교차 접근

Medium
근거
student_management_notes 정책이 for all using (is_teacher()) — 담당 교사가 아닌 임의의 교사 허용. listManagedStudents도 소속 무관하게 role='student' 전체를 조회.
영향
다중 교사 배포 시 아무 교사나 모든 학생의 birth_date·메모(미성년자 PII)를 읽고 수정 가능.
수정
담당 관계로 스코프: 0028의 is_teacher_of_student 조인 패턴을 정책에 적용, 목록도 교사의 반으로 한정.
SEC-4

교사 권한 승격 무차별 대입 가능

Medium
근거
claim_teacher(code)를 인증된 누구나 횟수 제한 없이 호출 가능. 레이트 리밋/락아웃 없음.
영향
teacher_code가 짧거나 추측 가능하면 학생이 교사로 승격 → 전체 제출·학생관리·상점 접근.
수정
긴 랜덤 코드 + 시도 제한(advisory-lock 지연 또는 사용자별 시도 카운터). 강점: 가입 트리거가 role을 student로 강제하고 코드는 서버 전용(app_config SELECT 정책 없음).
SEC-5

전체 사용자 명단 · 아이디 열거

Low
근거
profiles SELECT 정책이 using (true) — 인증된 누구나 전원의 id·display_name(=아이디)·role 조회.
영향
합성 이메일 로그인(id@flowpy.local)과 결합 시 유효 아이디 전량이 노출돼 크리덴셜 스터핑 표면 확대, 학생 명단 유출.
수정
SELECT를 필요한 행(본인 · 같은 반 · 담당 교사)으로 제한하거나, 이름은 스코프된 RPC/뷰로 노출.
SEC-6

임의 교사에게 쪽지 발송 (스팸)

Low
근거
messages insert 정책이 학생 → role='teacher' 전원 허용(담당 무관).
영향
학생이 소속 외 교사에게도 쪽지·푸시 유발 가능(스팸 벡터).
수정
수신자를 학생이 등록된 반의 담당 교사로 제한.
SEC-7

엣지 함수 잘못된 JSON → 500

Info
근거
send-push가 await req.json()을 try 없이 호출 → 잘못된 본문에 400 대신 미처리 500.
수정
파싱을 감싸 400 반환. 기능적 위험은 낮음.

04 강점

잘 설계된 보안 지점

  • 일관된 소유권 RLS — 모든 테이블에 RLS 활성화, created_by = auth.uid() 모델을 전 테이블에 일관 적용.
  • search_path 고정 — 모든 SECURITY DEFINER 함수가 search_path를 핀 고정(인젝션 차단).
  • 상점 결제 원자성 — decide_shop_order가 FOR UPDATE로 재고·잔액을 서버에서 원자 검증 → 위조 불가.
  • 포인트 원장 보호 — 자동 지급은 DB 트리거 경유, points_ledger 클라 직접 insert 차단(교사 또는 definer 함수만).
  • 푸시 서버 검증 — 엣지 함수가 수신자·문구를 서버에서 재도출하고 행 소유권 확인(스푸핑 방지), class_reminder는 service_role 토큰 요구.
  • XSS 정화 — 마크다운을 marked 후 DOMPurify.sanitize로 정화.
  • 포트폴리오 서브시스템(0028) — 정확한 teacher_id 스코프, 제출 스냅샷 불변 트리거, 스토리지 경로 소유권 체크, 낙관적 리비전 잠금.
  • RLS 재귀 회피 — is_teacher()·is_enrolled_in_class() definer 함수로 정책 상호참조 무한재귀 제거(0027).
